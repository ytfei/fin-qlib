# Routine 任务生成分析

## 为什么会产生 24 个任务？

### 1. 关键时间信息

从日志中提取：

```
当前时间: 2026-01-21 00:00:00
上次滚动测试开始时间: 2022-12-26 00:00:00
时间间隔: 744 天 (交易日)
滚动步长: 30 天
```

### 2. 任务数量计算

```
任务数量 = 时间间隔 / 滚动步长
        = 744 天 / 30 天
        ≈ 24.8
        → 24 个任务
```

### 3. 为什么需要这么多任务？

**Rolling Strategy（滚动策略）** 的核心思想是：
- 每隔一段时间（滚动步长）重新训练一个新模型
- 使用最新的数据来适市场变化
- 旧模型逐渐被淘汰

**时间线示意**：

```
2022-12                    2026-01
  |───────────────────────────|
  |
  上次训练
  (模型 A)

  |──30天──> ──30天──> ... ──30天──>
    训练      训练             训练
   模型 B    模型 C   ...    模型 Y (24个)
```

### 4. 每个任务的结构

每个任务包含完整的数据分段：

```python
# 任务 1 (示例)
{
    "model": {...},
    "dataset": {
        "segments": {
            "train": [2020-01-01, 2022-01-01],  # 训练集：2年数据
            "valid": [2022-01-01, 2022-07-01],  # 验证集：6个月
            "test":  [2022-12-26, 2023-01-25]   # 测试集：30天（滚动）
        }
    }
}

# 任务 2
{
    "dataset": {
        "segments": {
            "train": [2020-02-25, 2022-02-25],  # 整体向后滚动
            "valid": [2022-02-25, 2022-08-25],
            "test":  [2023-01-25, 2023-02-24]   # 新的测试期
        }
    }
}

# ... 任务 3-24 类似
```

### 5. 滚动模式 (ROLL_SD vs ROLL_EX)

#### ROLL_SD (Sliding Window) - 滑动窗口
```python
# 配置
rolling_config:
    step: 30
    rtype: "ROLL_SD"

# 效果：所有分段向后移动，保持窗口大小不变
train: [2020-01-01, 2022-01-01]  → [2020-02-25, 2022-02-25]
valid: [2022-01-01, 2022-07-01]  → [2022-02-25, 2022-08-25]
test:  [2022-12-26, 2023-01-25]  → [2023-01-25, 2023-02-24]
```

#### ROLL_EX (Expanding Window) - 扩展窗口
```python
# 配置
rolling_config:
    step: 30
    rtype: "ROLL_EX"

# 效果：训练集不断扩大，valid 和 test 滚动
train: [2020-01-01, 2022-01-01]  → [2020-01-01, 2022-02-25]  # 扩展
valid: [2022-01-01, 2022-07-01]  → [2022-02-25, 2022-08-25]  # 滚动
test:  [2022-12-26, 2023-01-25]  → [2023-01-25, 2023-02-24]  # 滚动
```

### 6. 代码执行流程

```python
# qlib/workflow/online/strategy.py:167-189
def prepare_tasks(self, cur_time):
    # 1. 获取当前在线的模型
    latest_records, max_test = self._list_latest(self.tool.online_models())
    # max_test = 2022-12-26 (上次任务的测试开始时间)

    # 2. 计算时间间隔
    calendar_latest = transform_end_date(cur_time)  # 2026-01-21
    interval = self.ta.cal_interval(calendar_latest, max_test[0])  # 744 天

    # 3. 生成后续任务
    for rec in latest_records:
        task = rec.load_object("task")  # 加载上次任务配置
        res.extend(self.rg.gen_following_tasks(task, calendar_latest))
        # RollingGen.gen_following_tasks() 会循环生成任务，直到测试段超过当前时间

    return res  # 返回 24 个新任务
```

### 7. 这些任务是什么？

**是的，这些是训练任务！**

每个任务都会：
1. 加载指定时间段的数据
2. 训练一个新的 LightGBM 模型
3. 在测试集上生成预测
4. 将预测保存到 MLflow Recorder
5. 标记为 "online" 状态

### 8. 为什么第一次运行没有这么多任务？

**first_run.py** (首次训练)：
- 只训练 1 个初始模型
- 使用配置文件中的初始 segments
- 任务数量：1 个

**run_routine.py** (日常更新)：
- 检查上次训练时间到当前时间的间隔
- 根据滚动步长补齐所有中间的任务
- 任务数量：间隔天数 / 滚动步长

### 9. 如何减少任务数量？

#### 方法 1: 增加滚动步长
```yaml
# config/online_config.yaml
rolling_config:
    step: 90  # 从 30 改为 90 天，任务数减少 67%
```

#### 方法 2: 限制最大滚动次数
在代码中添加：
```python
# 在 RollingStrategy.prepare_tasks() 中
max_tasks = 10  # 最多生成 10 个任务
if len(res) > max_tasks:
    res = res[:max_tasks]
```

#### 方法 3: 使用固定模型（不滚动）
```yaml
rolling_config:
    enabled: false  # 禁用滚动
```

### 10. 24 个任务的执行时间

根据日志：
```
train tasks: 4% | 2/24 [01:48<19:52, 54.21s/it]
```

- 平均每个任务：~54 秒
- 24 个任务总时间：54 × 24 = 1296 秒 ≈ 21.6 分钟

## 总结

| 问题 | 答案 |
|------|------|
| **为什么 24 个任务？** | 从上次训练(2022-12)到现在(2026-01)有744天，每30天滚动一次 |
| **这些任务是什么？** | 训练任务，每个训练一个新的 LightGBM 模型 |
| **为什么会这样？** | Rolling Strategy 的设计：定期用新数据训练新模型 |
| **如何减少任务？** | 增加滚动步长 (step: 30→90) 或禁用滚动 |
| **执行时间多久？** | 约 20-25 分钟 (24任务 × 54秒/任务) |

**建议**：
- 如果只是测试，将 `step` 改为 90 或 180 天
- 如果是生产环境，保持 30 天以获得更好的模型更新频率
- 如果想要更快结果，考虑使用 TrainerRM 并行训练
